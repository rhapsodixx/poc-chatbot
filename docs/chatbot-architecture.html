🏗️ System Architecture Overview

The chatbot follows a classic RAG (Retrieval-Augmented Generation) architecture with three Docker-composed services. The frontend communicates with the FastAPI backend, which orchestrates retrieval from ChromaDB and generation via OpenRouter.

📥 Ingestion Pipeline

The ingestion pipeline (backend/app/ingestion/pipeline.py) is an async process that crawls the satusatu.com sitemap, cleans HTML, chunks content semantically, and upserts embeddings into ChromaDB. It supports incremental updates via content-hash diffing.

Step 1 — Sitemap Parsing

1

fetch_sitemap(sitemap_url)

Fetches the XML sitemap using httpx.AsyncClient with redirect-following. Supports sitemap index files — if the root sitemap contains <sitemap> elements, it recursively fetches each sub-sitemap. Returns a Dict[str, str] mapping each URL to its <lastmod> timestamp. Handles both namespaced and non-namespaced XML.

📄 Key Code — fetch_sitemap 
async def fetch_sitemap(sitemap_url: str) -> dict[str, str]:
    urls: dict[str, str] = {}
    namespaces = {"sm": "http://www.sitemaps.org/schemas/sitemap/0.9"}

    async with httpx.AsyncClient(follow_redirects=True) as client:
        response = await client.get(sitemap_url)

    root = ET.fromstring(response.text)
    # Check for sitemap index → recursive fetch
    sitemap_refs = root.findall(".//sm:sitemap/sm:loc", namespaces)
    if sitemap_refs:
        for ref in sitemap_refs:
            sub_urls = await fetch_sitemap(ref.text.strip())
            urls.update(sub_urls)
        return urls

Step 2 — Page Fetching & HTML Cleaning

2

fetch_and_clean_page(url, client, lastmod)

Concurrency is controlled via asyncio.Semaphore(max_concurrent=5). Each page is fetched with a shared httpx.AsyncClient for connection reuse. HTML parsing is offloaded to a background thread via asyncio.to_thread() to avoid blocking the event loop.

HTML Cleaning Strategy

  • Decompose non-content tags: script, style, nav, header, footer, aside, noscript, iframe
  • Extract og:image meta tag (fallback: first <img> tag)
  • Prefer <main> → <article> → <body> for content extraction
  • Collapse whitespace: 3+ newlines → 2, 2+ spaces → 1
  • Skip pages with <50 characters of cleaned content
  • Infer page type from URL patterns (attraction, ticket, blog, faq, etc.)

Step 3 — Semantic Chunking

3

chunk_text(page, max_chunk_size=256, overlap=32)

A two-tier chunking strategy optimized for embedding quality:

Chunking Algorithm

  1. Primary split — by double-newlines (\n\n) or markdown headings (# ## ###)
  2. Accumulate sections until word count exceeds max_chunk_size (256 words)
  3. Oversized sections — fallback to sentence-level splitting via _split_by_sentences()
  4. Overlap — 32-word overlap between consecutive chunks ensures context continuity at boundaries

Each chunk carries rich metadata: url, title, page_type, meta_description, image_url, lastmod, and content_hash.

Step 4 — ChromaDB Upsert

4

upsert_chunks(chunks) — Global Batched Upsert

All chunks from all updated pages are collected globally and upserted in batches of 50. Before upserting, old chunks for updated URLs are deleted via delete_by_url(). Stale URLs (present in ChromaDB but removed from sitemap) are cleaned up at the end. Chunk IDs are deterministic MD5 hashes of url + text[:50].

🤖 RAG Pipeline — 3 Guardrail System

The real-time query pipeline (backend/app/services/rag.py) implements a 3-step guardrail architecture to ensure domain confinement, prevent hallucination, and enable graceful human handoff.

Guardrail 1 — Intent Classification

classify_intent(query) → "RELEVANT" | "ITINERARY" | "OFFTOPIC"

A lightweight LLM call classifies each incoming query using INTENT_CLASSIFICATION_PROMPT. The prompt asks the model to emit exactly one word. The system checks for the presence of keywords in the response to handle minor variations.

Fail-Open Strategy: If classification fails (LLM error, timeout), it defaults to "RELEVANT" — letting the downstream guardrails handle safety. This prevents classification errors from blocking legitimate queries.

Guardrail 2 — Semantic Retrieval & Confidence Check

retrieve_context(query) → (context, best_score, sources)

Queries ChromaDB for the top max_retrieved_chunks + 3 results (extra to compensate for filtered homepage chunks). ChromaDB returns cosine distances (0 = identical, 2 = opposite), converted to similarity:

similarity = 1 - (distance / 2)

Homepage Filtering: Chunks from root URLs (ending in /en-US, /id-ID, .com, .com/) are filtered out to prevent the LLM from extracting products without proper catalog URLs or images.

Confidence Gate: If no chunks pass the similarity_threshold (default: 0.35), the system bypasses generation entirely and triggers a human handoff with randomized friendly messages.

Guardrail 3 — Conditioned Generation

generate_answer(query, context, intent) → (response, usage)

The LLM is given a strictly constrained system prompt with the retrieved context injected. Two system prompts exist:

  • GENERATION_SYSTEM_PROMPT — for general product/attraction queries
  • ITINERARY_SYSTEM_PROMPT — for trip planning (max 3 days, realistic timing)

Both prompts instruct the LLM to output TRIGGER_HANDOFF if the context is insufficient. The backend checks for this keyword and replaces it with a handoff payload containing Email and WhatsApp links.

Model Failover: If the primary model (Gemini 2.5 Flash) fails, the system automatically retries with the fallback model (GPT-4o-mini). If both fail, it returns TRIGGER_HANDOFF.

🧬 Embedding & Vector Search Deep Dive

Embedding Configuration

Embeddings are generated via ChromaDB's OpenAIEmbeddingFunction, configured to route through OpenRouter:

embedding_function = embedding_functions.OpenAIEmbeddingFunction(
    api_key=settings.openrouter_api_key,
    model_name=settings.openrouter_embedding_model,  # openai/text-embedding-3-small
    api_base=settings.openrouter_base_url,            # https://openrouter.ai/api/v1
)

collection = client.get_or_create_collection(
    name="satusatu_knowledge",
    metadata={"hnsw:space": "cosine"},  # HNSW index with cosine distance
    embedding_function=embedding_function,
)

Vector Search Mechanics

  • Index: HNSW (Hierarchical Navigable Small World) — approximate nearest neighbor, O(log N) query time
  • Distance Metric: Cosine distance (0 = identical, 2 = opposite)
  • Similarity Conversion: similarity = 1 - (distance / 2) — maps to [0, 1] range
  • Threshold: SIMILARITY_THRESHOLD=0.35 — chunks below this are discarded
  • Max Results: MAX_RETRIEVED_CHUNKS=5 — top-K after filtering

📝 Prompt Engineering

The system uses three carefully crafted prompt templates, each serving a specific role in the pipeline.

🔍 INTENT_CLASSIFICATION_PROMPT

Purpose: Fast, single-word classification gate.

Outputs: RELEVANT | ITINERARY | OFFTOPIC

Design Decision: Kept minimal to minimize token usage and latency. No system prompt — the classification instruction is the user message itself.

"Is this query regarding a product, ticket, or itinerary
on satusatu.com? Reply with EXACTLY one word:
RELEVANT, ITINERARY, or OFFTOPIC"
💬 GENERATION_SYSTEM_PROMPT (General Queries)

Key Design Decisions:

  • Structured JSON Output: Forces the LLM to output a ```json block with a strict schema for product cards. This enables the frontend to render rich UI components instead of plain text.
  • Product Validation: Products must have a catalog-specific source URL (containing /catalog/). Prevents extraction of products from homepage listings where images/URLs may be incomplete.
  • Minimum Sold Threshold: Products with <100 units sold are excluded from "best"/"recommended" queries to prevent promoting untested items.
  • Tag Extraction: "locally curated" must be an exact string match in Context — not inferred. "Kids friendly" etc. can be inferred from product descriptions.
  • TRIGGER_HANDOFF: Magic string that the backend intercepts to trigger the human handoff UI.
🗺️ ITINERARY_SYSTEM_PROMPT (Trip Planning)

Additional Constraints vs. General Prompt:

  • Max 3 Days: If user requests >3 days, decline with a friendly apology — no JSON output.
  • Realistic Timing: Must estimate activity durations and commute times between locations.
  • Locally Curated Priority: Must prefer "locally curated" products when building itineraries.
  • Day-by-Day JSON: Output schema includes itinerary[].day, activities[].time, activities[].products[] with full product card data.

🚀 Suggestions for Improvement

Prioritized improvements for evolving from MVP to production-grade.

1

Streaming Responses (SSE)

Currently, the entire LLM response is awaited before sending to the client. Implementing Server-Sent Events would deliver tokens in real-time, dramatically improving perceived latency. FastAPI supports StreamingResponse natively.

2

Conversation Memory

No multi-turn context exists — each message is processed independently. Adding session-based chat history (Redis or in-memory with TTL) would enable follow-up questions like "tell me more about that" or "what about the pricing?".

3

Hybrid Search (Semantic + BM25)

Pure semantic search can miss exact keyword matches (e.g., specific product names). Combining with BM25/keyword search and using Reciprocal Rank Fusion (RRF) would improve recall for both semantic and lexical queries.

4

Cross-Encoder Re-Ranking

Add a re-ranking stage (e.g., Cohere Rerank, BGE Reranker) after initial bi-encoder retrieval. This refines the top-K results with much higher accuracy, improving answer quality with minimal latency cost.

5

Evaluation Framework (RAGAS)

Implement automated evaluation using RAGAS (Retrieval-Augmented Generation Assessment) to measure faithfulness, answer relevancy, and context precision. Essential for data-driven prompt tuning.

6

Query & Response Caching

Cache frequent or semantically similar queries to skip LLM calls entirely. Use a TTL-based cache (Redis) keyed on query embedding similarity. Could reduce LLM costs by 30-50% for common questions.

7

Observability (OpenTelemetry + LangSmith)

Add structured logging, distributed tracing, and LLM observability (LangSmith or Langfuse) to monitor retrieval quality, prompt performance, latency distribution, and error rates in production.

8

Webhook-Based Ingestion

Replace scheduled re-ingestion with CMS webhook triggers to auto-ingest when product pages are updated. Reduces stale content window from hours to seconds.

9

Rate Limiting & API Authentication

Add API key authentication and rate limiting (per-IP and per-key) to protect against abuse. Use FastAPI middleware with SlowAPI or a reverse proxy (Nginx/Caddy) for rate limiting.

10

Chunk Overlap Tuning

Current overlap (32 words) is a reasonable default but hasn't been empirically optimized. Run A/B tests with different overlap ratios (10%, 15%, 25%) and measure retrieval recall using a ground-truth Q&A dataset.
